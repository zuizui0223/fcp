# Journal of Biogeography archive and release protocol

This protocol separates three objects that must not be conflated:

1. the **journal submission package** uploaded to the manuscript system;
2. the **permanent repository release** cited for code, analytical data and provenance; and
3. the **GBIF citation objects** described in `docs/jbi_gbif_doi_protocol.md`.

## 1. Complete author-controlled fields

Before freezing a release, complete and approve:

- `docs/jbi_author_confirmation_form.md`;
- `docs/jbi_title_page_template.md`;
- `docs/jbi_cover_letter_template.md`;
- the Data Accessibility Statement in the anonymized manuscript;
- final taxon-image selection and rights record.

Do not replace an unknown procedure with a plausible-sounding retrospective description.

## 2. Obtain GBIF citation identifiers

Follow `docs/jbi_gbif_doi_protocol.md`.

- Submit `jbi_gbif_broad_download_request.json` through an authenticated GBIF account.
- Archive `jbi_gbif_exact_occurrence_subset.csv.gz` at a permanent public URL.
- Register the exact subset as a GBIF Derived Dataset using the 389 parent-dataset contribution counts.
- Insert the broad-download and Derived Dataset citations in the manuscript, title page and cover letter.

## 3. Freeze a submission commit

1. Ensure the submission-package validator passes.
2. Ensure the GBIF DOI-bundle validator passes.
3. Run the release builder in strict mode.
4. Review every file in the generated submission ZIP.
5. Commit any corrections and repeat validation.
6. Record the final commit SHA in the author confirmation form.

Commands:

```bash
python validate_jbi_submission_package.py
python validate_jbi_gbif_doi_bundle.py
python prepare_jbi_submission_release.py --strict --output-dir analysis_outputs/jbi_release
```

Strict mode fails when author-facing submission files retain `Not verified` markers or bracketed placeholders.

## 4. Create the permanent code/data release

Preferred route:

1. connect the GitHub repository to Zenodo or another approved archival service;
2. create a versioned GitHub release from the frozen submission commit;
3. archive that exact release;
4. verify the archived commit or source bundle against the release manifest;
5. insert the resulting DOI into all submission files.

The journal submission ZIP generated by the release builder is not a substitute for archiving the repository release.

## 5. Zenodo metadata

Use `docs/jbi_zenodo_metadata_template.json` as a preparation aid. Replace every `REPLACE_...` value with author-approved metadata before creating a deposition. Confirm:

- creator order matches final authorship;
- ORCIDs and affiliations are exact;
- title matches the manuscript;
- description does not overstate causality or phylogenetic certainty;
- licence is approved for the deposited code and data;
- related identifiers include the manuscript DOI after publication when available.

## 6. Submission ZIP contents

The release builder packages journal-facing and audit files, including:

- anonymized manuscript;
- separate title page;
- cover letter;
- Supporting Tables S1–S17;
- Appendix S1;
- figures and phylogenetic trees;
- GBIF exact-subset and DOI-preparation bundle;
- provenance and validation documents.

The generated manifest records the source commit, mode, file sizes and SHA-256 values.

## 7. Final checks

Before upload:

- open the generated ZIP and inspect every file;
- confirm that the anonymized manuscript contains no author identifiers;
- confirm that the title page contains the approved author metadata;
- confirm all DOI strings are final and consistent;
- confirm the exact taxon image file and licence evidence are retained outside the ZIP when the journal requests a separate upload;
- rerun strict validation on the frozen commit;
- retain the validation artifact and release manifest with the submission record.
